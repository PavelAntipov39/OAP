# Правила работы с задачами

## Что такое задача

Задача — это атомарная единица работы, которую выполняет агент или человек. Каждая задача имеет чёткую цель, критерии приемки и фиксированные роли участников.

---

## Структура задачи

Каждая задача должна содержать:

- **Цель** — конкретное, измеримое описание того, что нужно достичь
- **Ожидаемый результат** — конкретный исход, по которому можно судить о выполнении
- **Критерии приемки** — условия, при которых задача считается выполненной
- **Целевые артефакты** — файлы, документы, изменения кода, которые должны появиться
- **Причина приоритета** — обоснование, почему задача важна именно сейчас
- **Оперативная память** — рабочие заметки/факты по текущей задаче (`context_package.operational_memory[]`)
- **План подключения агентов** — анализ, кого подключать как соисполнителей (`context_package.collaboration_plan`)
- **План A/B** (если применимо) — окно теста и критерии pass/rollback (`context_package.ab_test_plan`)

---

## Роли в задаче

Участниками задачи могут быть агенты или люди.

| Роль | Кто это | Обязанности |
|------|---------|-------------|
| **Постановщик** | Агент или человек, создавший задачу | Формулирует цель, заполняет критерии приемки, прикладывает контекст и доказательства |
| **Исполнитель** | Основной ответственный за выполнение | Берёт задачу в работу, отвечает за результат, обновляет статус |
| **Соисполнители** | Дополнительные участники | Выполняют свою часть работы, отвечают за отдельные вопросы в рамках задачи |
| **Наблюдатель** | Заинтересованная сторона | Не участвует активно, но должен быть в курсе хода и может привлекаться для уточнений |

## Жизненный цикл задачи

```
backlog → ready → in_progress → ab_test → in_review → done
```

| Статус | Описание |
|--------|----------|
| `backlog` | Задача зафиксирована, ещё не готова к работе |
| `ready` | Готова к выполнению — все обязательные поля заполнены |
| `in_progress` | Исполнитель взял задачу в работу |
| `ab_test` | Идет проверка candidate-практики в рамках A/B-этапа |
| `in_review` | Результат передан на проверку |
| `done` / `completed` | Задача завершена, результат принят |

---

## Готовность задачи (Readiness)

Автоматическая проверка считает задачу готовой, если заполнены:

1. Цель задачи (`goal`)
2. Ожидаемый результат (`expected_outcome`)
3. Критерии приемки (`acceptance_criteria`)
4. Целевые артефакты (`target_artifacts`)
5. Доказательства или источник (`evidence_refs` / `origin_ref`)

**Итог:** `ready` — можно брать в работу, `needs_clarification` — нужно уточнение у постановщика.

Дополнительно к автоматической проверке существует ручное подтверждение ревьюером (`approved` / `needs_clarification` / `not_set`).

## Правила для агентов

### При создании задачи

- Заполнять: цель, ожидаемый результат, критерии приемки, целевые артефакты
- Указывать `source_agent_id` (постановщик) и `executor_agent_id` (исполнитель)
- Прикладывать `evidence_refs` — ссылки на данные, обосновывающие задачу
- Выбирать тип происхождения: `improvement`, `recommendation` или `telemetry`
- Заполнять `context_package` — пакет контекста для исполнителя
- Для analyst-pipeline v2 дополнительно заполнять:
  - `operational_memory[]`: `{ key, title, value, source_ref?, updated_at? }`
  - `collaboration_plan`: `{ analysis_required, suggested_agents[], selected_agents[], rationale, reviewed_at? }`
  - `ab_test_plan` (для candidate-практик): `{ enabled, sessions_required, pass_rule, target_metric, expected_delta_pct, guardrails[], rollback_on_fail }`

### При работе с задачей

- Прочитать `context_package` перед началом работы
- Соблюдать `mandatory_rules` из контекст-пакета
- Перед стартом исполнения провести `collaboration analysis` и актуализировать `collaboration_plan`
- Записывать события в timeline: тип события, статус до и после, действующий агент
- Обновлять статус строго по факту, не авансом
- При необходимости привлекать соисполнителей и информировать наблюдателей
- Для A/B задач:
  - вести checkpoint-события `ab_test_checkpoint`;
  - считать pass по правилу `target_plus_guardrails`:
    `median(target_delta_pct) >= expected_delta_pct` и `guardrail_breached_count = 0`;
  - при провале фиксировать `ab_test_failed` и `rollback_applied`.

### При завершении задачи

- Убедиться, что все целевые артефакты созданы или изменены
- Перевести статус в `in_review` для проверки
- Зафиксировать финальное событие в timeline с описанием результата

## Приоритеты

| Приоритет | Значение |
|-----------|----------|
| `high` | Критично — влияет на работу системы или блокирует других |
| `medium` | Важно — нужно сделать в текущем цикле |
| `low` | Желательно — можно отложить без серьёзных последствий |
